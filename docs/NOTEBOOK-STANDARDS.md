# Notebook Authoring Standards

The quality contract for every notebook in `chapters-companion/`. Phase 2 authors (human or agent) follow
this so 100+ notebooks read as one consistent, trustworthy course — not a pile of scripts.

The North Star, borrowed from the book's playbook: **the reader must feel they're learning
from a seasoned expert who also teaches brilliantly.** A notebook that merely *runs* isn't
done; it has to *teach*.

---

## 1. Notebook types

| Type | Purpose | Shape |
|---|---|---|
| **concept-lab** | Make an abstract idea tangible by running it | Many small cells, lots of 🔮 predict-then-run, mostly offline |
| **walkthrough** | Build one mechanism end-to-end in isolation | Builds up a working thing; ends pointing at the blueprint |
| **drill** | Repetition to build fluency (typing, async, Big-O, SQL) | Short tasks + checks; minimal narration |
| **worksheet** | Reflection/planning (career, system-design, ADRs) | Prompts + fill-in cells; little/no code |

---

## 2. The standard notebook skeleton

Every notebook has these cells, in order:

1. **Title + header (markdown).** Chapter/section it maps to, type badge, and a one-line
   promise. Include the standard banner:
   > 📓 *Companion to* **Modern Agentic AI Engineer** *· Ch NN §x · type: walkthrough*
2. **🧠 Why this matters (markdown).** 3–5 sentences of mental model before any code.
3. **Objectives + prereqs (markdown).** "By the end you can…" and what to run first.
4. **Setup (code).** Imports, `load_dotenv()`, a `MOCK = os.getenv("COMPANION_MOCK", "1")`
   switch, and a one-cell environment check. Never hardcode keys.
5. **Body (alternating markdown ↔ code).** Small steps. Before a meaningful output, a
   🔮 *predict* prompt. After it, a short "what you just saw." Surface the chapter's ⚠️
   pitfalls inline where they bite.
6. **🎯 Senior lens (markdown).** The trade-off/judgment a senior applies here.
7. **Recap (markdown).** 3–5 bullets of what was learned.
8. **Exercises (markdown + empty code cells).** 2–4 tasks that *change* something and predict
   the result. Solutions go in `solutions/` (Phase 2), not inline.
9. **Next (markdown).** Link to the next notebook, the relevant blueprint, and the capstone
   dir this feeds.

---

## 3. Reproducibility & cost (non-negotiable)

- **Offline-first.** Prefer tiny local computation. When a model call is essential, gate it:
  - `MOCK=1` (default) returns a **canned, realistic response** so the notebook runs free and
    deterministically in CI and for readers without a key.
  - `MOCK=0` hits the live API. Document the approximate **token cost** in the setup cell.
- **Seeds set** for anything stochastic; call out where determinism isn't possible (LLMs).
- **Pinned deps.** Notebooks rely only on `requirements.txt`; if a notebook needs extra
  packages, it declares them in its first code cell and in the chapter `PLAN.md`.
- **Tiny committed fixtures.** Sample data lives in the chapter's `data/`, kept small. Larger
  data is generated by a cell, never committed as a binary blob.
- **Default to the latest, most capable Claude models** in examples (the book's stack is
  Anthropic-first), with provider-agnostic patterns where the chapter is about portability.

---

## 4. Safety & secrets

- Secrets come **only** from environment (`.env`, git-ignored). The setup cell fails fast
  with a friendly message if a required key is missing in `MOCK=0`.
- Destructive or costly operations (real cloud spend, browser control, web posting) are
  **opt-in**, clearly flagged ⚠️, and default to a dry-run/sandbox.
- No secrets, tokens, or personal data in committed outputs.

---

## 5. Output hygiene

- **Commit notebooks with cleared outputs** (CI re-executes them), *except* where a rendered
  output is itself the teaching artifact (a chart, a diagram) — then commit that cell's
  output deliberately and keep it small.
- Keep cell outputs short; truncate long dumps. A wall of logs teaches nothing.

---

## 6. CI verification (Phase 2)

- `nbval`/`papermill` execute every notebook in `MOCK=1` on push and on a schedule — green
  means "runs top-to-bottom, no errors."
- A nightly **gated** job runs a sampled subset with `MOCK=0` to catch SDK drift.
- The `capstone-project/` gets a separate `docker compose up` + healthcheck smoke test.
- Lint: `ruff`/`black` for code cells, `nbqa` to apply them to notebooks.

---

## 7. The "would a senior have written this?" checklist

Before a notebook is done:

- [ ] Mental model **before** mechanics; not a code dump.
- [ ] At least one 🔮 predict moment and one ⚠️ real pitfall.
- [ ] Code shapes match the book and the current SDKs.
- [ ] Runs free and clean in `MOCK=1`; live path documented.
- [ ] Ends with a recap, exercises, and links onward (blueprint/capstone).
- [ ] No AI-tells (see the book playbook §5): no throat-clearing openers, no "X is not Y;
      it's Z" on every cell. Direct, expert, warm.
